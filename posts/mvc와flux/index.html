MVC 패턴과 Flux 패턴

MVC 패턴이란?

MVC는 Model, View, Controller의 약자 입니다.
프로젝트의 구성과 기능들을 3가지의 역할로 나누어서 개발하는 패턴입니다.

애플리케이션의 구성 요소를 3가지 역할로 구분하여 각각의 구성 요소에만 집중하여 개발할 수 있습니다. 재사용성과 확장성이 용이하다는 장점이 있고, 애플리케이션이 복잡해질수록 모델과 뷰의 관계가 복잡해지는 단점이 있습니다.


  • 모델(Model)은 애플리케이션의 데이터인 데이터베이스, 상수, 변수 등을 뜻합니다.
  • 뷰(View)는 button, input, div 등 사용자 인터페이스 요소를 나타냅니다.
  • 컨트롤러(Controller)는 이벤트나 함수, 메인 로직을 담당하며 모델과 뷰를 잇는 다리 역할을 합니다.

예를 들어서 어떠한 유저가 글 작성 버튼을 눌렀을 때 Controller(글 생성)가 실행되며 Model(DB)가 변경되게 됩니다. 그리고 View(글 목록)가 변경되어 유저에게 글이 생성된 것을 보여줍니다. 하지만 웹 사이트의 규모가 점점 커지는 만큼 MVC 패턴으로 개발을 하면서 단점들이 나오기 시작합니다.


위 그림과 같이 Client에서는 Model(상태)와 View(화면)사이에서의 양방향 데이터 흐름으로 웹 어플리케이션의 규모가 점점 커진다면 어떠한 기능을 추가할 때마다 시스템의 복잡도는 기하급수적으로 증가하게 되며, 개발자 입장에서는 데이터의 예측과 파악이 어려워져 에러와 버그들을 쏟아내게 됩니다.

이러한 문제점들로 인해서 실제로 페이스북에서는 알림 기능이 제대로 작동하지 않아 개발자들이 어려움을 겪었습니다. 페이스북에서는 에러와 버그들의 대부분은 양방향 데이터의 흐름으로 인한 문제점이라고 결론을 내렸습니다.

그래서 2014년 페이스북은 F8 컨퍼런스에서 MVC 패턴을 기반으로 한 Flux 패턴이라는 새로운 아키텍처를 발표합니다. 점점 거대해져 가는 웹 서비스의 데이터 흐름을 일관성 있게 관리하며 개발자가 예측 가능하도록 하는 아키텍처 입니다.

Flux 패턴


예를 들어서 위 그림과 같이 View(화면)에서 Action(글 생성 클릭)이 발생하여 Dispatcher(함수)가 Store(DB)을 변경한 후 View(화면)가 렌더링 되게 됩니다. 이렇게 MVC패턴과 달리 View가 직접 Model을 바꾸는 것이 아니라 Action과 Dispatcher에 의해서 Store을 거쳐 변경 되게 됩니다. 양방향 데이터의 흐름에서 단방향 데이터의 흐름으로 바뀐 Flux 패턴의 모습입니다.

리액트는 이러한 단방향 데이터의 흐름을 지향하며 상태 관리 라이브러리들은 이런 Flux 패턴을 사용하였습니다.

이제 모든 데이터들은 Store을 거쳐가게 되며 개발자들은 Store만 보게 된다면 데이터의 흐름을 한눈에 파악하기 쉽습니다.

결론

  • Flux 패턴은 MVC 패턴의 단점을 보완하여 나온 아키텍처이며, 단방향 데이터의 흐름이다.
  • 단방향 데이터의 흐름은 개발자가 예측 가능할 수 있게 하며 데이터를 일관성 있게 관리 할 수 있다.
  • Flux 패턴은 데이터의 복잡성을 줄여주며 개발자가 전체적인 코드를 파악하는 것에 도움이 된다.
